# Practice Session Schema

A **session** is an ordered list of exercise segments that concatenate into
one sloppak, with a FeedBack section marker at the start of each segment.
Sessions are generated by `buildSessionChart(session, baseCfg)` in `screen.js`
and launched through the same `POST /temp-sloppak` route as single exercises.

---

## Top-level session object

```json
{
  "version": 1,
  "name": "ii–V–I Workshop",
  "description": "Human-readable description shown in the UI.",
  "stringSetup": "guitar_6_standard",
  "tags": ["jazz", "intermediate"],
  "bpmLadder": {
    "enabled": false,
    "bpmStart": 80,
    "bpmTarget": 120,
    "bpmStep": 5,
    "repsPerStep": 1
  },
  "keyCycle": {
    "enabled": false,
    "strategy": "circle_of_fourths",
    "keysPerSession": 4
  },
  "segments": []
}
```

### Top-level fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `version` | int | 1 | Schema version — always 1 for now |
| `name` | string | required | Display name |
| `description` | string | `""` | Optional description |
| `stringSetup` | string | `"guitar_6_standard"` | Key into `STRING_SETUPS` |
| `tags` | string[] | `[]` | Freeform tags for filtering |
| `bpmLadder` | object \| null | null | Session-level BPM ladder (applied to every segment unless overridden) |
| `keyCycle` | object \| null | null | Session-level key cycling (applied to every segment unless overridden) |
| `segments` | Segment[] | required | Ordered list of exercise segments |

---

## BPM ladder config

When `enabled: true`, the segment's exercise is generated once at each BPM
step from `bpmStart` to `bpmTarget` (inclusive), with `repsPerStep` repetitions
at each step. A sub-section marker labels each BPM step (e.g., "80 BPM",
"85 BPM").

```json
{
  "enabled": true,
  "bpmStart": 60,
  "bpmTarget": 100,
  "bpmStep": 5,
  "repsPerStep": 1
}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `enabled` | bool | `false` | Whether to apply the ladder |
| `bpmStart` | int | 60 | Starting BPM |
| `bpmTarget` | int | 100 | Ending BPM (inclusive) |
| `bpmStep` | int | 5 | BPM increment per step |
| `repsPerStep` | int | 1 | Number of repetitions at each BPM before stepping up |

A session-level `bpmLadder` applies to all segments. A segment-level `bpmLadder`
overrides the session-level one for that segment only. Set `{ "enabled": false }`
on a segment to opt out of the session-level ladder.

---

## Key cycle config

When `enabled: true`, the segment's exercise is repeated `keysPerSession` times,
cycling through the keys in `strategy` order starting from the segment's `key`.
A sub-section marker labels each key.

```json
{
  "enabled": true,
  "strategy": "circle_of_fourths",
  "keysPerSession": 4
}
```

| Field | Type | Options | Default | Description |
|-------|------|---------|---------|-------------|
| `enabled` | bool | — | `false` | Whether to apply key cycling |
| `strategy` | string | `"circle_of_fourths"` `"circle_of_fifths"` `"chromatic"` `"random"` | `"circle_of_fourths"` | Key order |
| `keysPerSession` | int | 2–12 | 4 | How many keys to generate |

## Segment object

```json
{
  "id": "gt_thirds",
  "name": "Guide tones — 3rds",
  "kind": "guide_tones",
  "bpmLadder": null,
  "keyCycle": null,
  "config": {
    "key": "C",
    "scale": "major",
    "progression": "ii-V-I",
    "chordDepth": "seventh",
    "voices": "thirds_only",
    "bpm": 60,
    "bars": 8,
    "meter": "4/4",
    "subdivision": "quarter",
    "fretboardSystem": "caged",
    "shape": "E",
    "direction": "up_down",
    "sequence": "none",
    "keyCycle": "none"
  }
}
```

### Segment fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `id` | string | required | Unique identifier within the session |
| `name` | string | required | Section marker label shown in FeedBack's player |
| `kind` | string | required | Exercise type — see valid values below |
| `bpmLadder` | object \| null | `null` | Segment-level override; inherits session `bpmLadder` if null |
| `keyCycle` | object \| null | `null` | Segment-level override; inherits session `keyCycle` if null |
| `config` | object | required | Exercise configuration — all fields that `buildSingleChart` consumes |

### Valid `kind` values

| Kind | Generator | Description |
|------|-----------|-------------|
| `scale` | `buildScaleExercise` | Scale run in a shape/position |
| `chord_scales` | `buildChordScaleExercise` | Scale passages over a chord progression |
| `diatonic_arpeggios` | `buildArpeggioExercise` | Diatonic chord arpeggios through a progression |
| `progression_arpeggios` | `buildArpeggioExercise` | Arpeggio patterns over a named progression |
| `sweep_arpeggios` | `buildSweepArpeggioExercise` | CAGED-anchored sweep patterns |
| `chromatic` | `buildChromaticExercise` | Chromatic warmup patterns (1234, spider, etc.) |
| `guide_tones` | `buildGuideTonesExercise` | 3rds and/or 7ths voice-led through a progression |
| `modal_vamp` | `buildScaleExercise` | Single chord/scale area, no progression changes |
| `strum_comp` | `buildStrumCompExercise` | Held voiced chord (CAGED grip) struck on a strum pattern — rhythm-hand/comping drill (guitar only). Config: `strumPattern` (a `STRUM_PATTERNS` key), `voicingPosition` (`open`/`movable`). |

> The valid `kind` list above is illustrative, not exhaustive — a segment `kind` may be **any** registered practice type (the same set the `practiceType` select offers). See `screen.js` `buildSingleChart` for the full dispatch.

## Config field reference

All fields below live inside `segment.config`. Defaults apply when the field
is absent. Shape-aware systems (`caged`, `3nps`, `open`) will auto-resolve
`fretMin`/`fretMax` from `key + fretboardSystem + shape` — the raw fret fields
are ignored for those systems.

### Common fields (all kinds)

| Field | Type | Default | Options |
|-------|------|---------|---------|
| `key` | string | `"C"` | `"C"` `"C#"` `"Db"` `"D"` ... all 12 pitch classes |
| `scale` | string | `"major"` | See scale table below |
| `bpm` | number | `80` | 30–260 |
| `meter` | string | `"4/4"` | `"4/4"` `"3/4"` `"6/8"` `"5/4"` `"7/8"` |
| `subdivision` | string | `"eighth"` | `"quarter"` `"eighth"` `"sixteenth"` `"triplet"` |
| `bars` | int | `4` | 1–32 |
| `direction` | string | `"up_down"` | `"ascending"` `"descending"` `"up_down"` `"down_up"` `"random"` |
| `fretboardSystem` | string | `"caged"` | `"caged"` `"3nps"` `"open"` `"position"` `"full_neck"` `"single_string"` |
| `shape` | string | `"E"` | `"C"` `"A"` `"G"` `"E"` `"D"` (CAGED); `1`–`7` (3NPS) |
| `fretMin` | int | `0` | Used only when `fretboardSystem` is `"position"` or `"single_string"` |
| `fretMax` | int | `5` | Used only when `fretboardSystem` is `"position"` or `"single_string"` |
| `keyCycle` | string | `"none"` | `"none"` `"circle_of_fourths"` `"circle_of_fifths"` `"chromatic"` `"random"` |

### Scale / arpeggio fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `sequence` | string | `"none"` | `"none"` `"fours"` `"triplets"` `"thirds"` `"broken_triads"` `"yngwie_sixes"` |
| `repeatCount` | int | `1` | How many times to repeat the phrase before moving to the next section |
| `chordDepth` | string | `"seventh"` | `"triad"` `"seventh"` |
| `progression` | string | `"diatonic"` | Key into `COMMON_PROGRESSIONS` |
| `chordOverride` | string | `"auto"` | `"auto"` or any key in `CHORD_FORMULAS` |
| `chordScaleStrategy` | string | `"mode_of_moment"` | `"mode_of_moment"` `"chord_tone_emphasis"` |

### Guide tones (`kind: "guide_tones"`)

| Field | Type | Default | Options | Description |
|-------|------|---------|---------|-------------|
| `voices` | string | `"thirds_only"` | `"thirds_only"` `"sevenths_only"` `"both_alternating"` | Which guide tone(s) to generate per chord |
| `progression` | string | `"ii-V-I"` | any `COMMON_PROGRESSIONS` key | Chord sequence to apply guide tones over |
| `chordDepth` | string | `"seventh"` | `"triad"` `"seventh"` | Determines whether 7ths are available |

**Voice-leading rule:** each guide tone is placed at the MIDI pitch closest
to the previous note, constrained to the resolved fret range. Starting pitch
targets mid-range guitar (MIDI 64 = E4).

### Chromatic (`kind: "chromatic"`)

| Field | Type | Default | Options |
|-------|------|---------|---------|
| `chromaticPattern` | string | `"1234"` | `"1234"` `"4321"` `"1324"` `"1342"` `"2413"` |
| `fretMin` | int | `1` | Starting fret |
| `fretMax` | int | `4` | End fret (pattern shifts by one fret per string) |

### Modal vamp (`kind: "modal_vamp"`)

Same as `scale` — `buildScaleExercise` handles it. The difference is semantic:
the section label and description reflect a single sustained chord area rather
than a scale run over changes. Set `progression: "none"` or omit it.

## How concatenation works

`buildSessionChart(session, baseCfg)` iterates segments and applies time offsets:

```
segment 1: notes [0..dur1), sections [0], beats [0..dur1)
segment 2: notes [dur1..dur1+dur2), sections [dur1], beats [dur1..dur1+dur2)
...
```

Template IDs in `chordTemplates` / `handShapes` are offset per segment so
they stay globally unique across the combined chart.

`generateSession(session, baseCfg)` wraps `buildSessionChart` the same way
`generateExercise` wraps `buildSingleChart` — it adds the beats for the full
duration and returns the standard `{ version, session, chart }` exercise object.

---

## BPM ladder sub-concatenation

When a segment has `bpmLadder.enabled: true`, `buildBpmLadderChart(cfg, ladder)`
is called instead of `buildSingleChart`. It generates the same exercise at
BPM 60, 65, 70, … 100 (for bpmStart=60, bpmTarget=100, bpmStep=5), each run
getting its own sub-section marker ("60 BPM", "65 BPM", …).

The beats for each BPM step are generated at the correct tempo for that step
and concatenated, so the FeedBack transport stays accurate through tempo changes.

## Adding a custom session

Sessions live in JavaScript objects matching the schema above. To add a new
built-in session:

1. Add an entry to `BUILT_IN_SESSIONS` in `screen.js`.
2. Add a corresponding `<option>` to the session select in `screen.html`.
3. No backend changes needed.

Custom user sessions will eventually be saveable through the preset CRUD
routes (`POST /api/plugins/virtuoso/presets`) — the session object serializes
cleanly to JSON and can be stored alongside single-exercise presets.
